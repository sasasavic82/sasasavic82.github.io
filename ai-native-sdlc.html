The Problem

Why the Traditional SDLC is Broken

AI agents eliminated the code production bottleneck. They exposed a far more dangerous one: unconstrained logical assumptions compounding at machine speed.

🧨

Assumption Propagation

Each generation step inherits unchallenged assumptions from every previous step. By step five, the codebase contains dozens of compounded assumptions — internally consistent but externally wrong.

👁️

Code Review Collapse

A 2,000-line PR generated in 10 minutes takes 2 hours to review. The reviewer reconstructs reasoning from output. The agent has an information advantage over its own reviewer.

🔄

The Vicious Cycle

Iterative "improvements" cause the model to bloat the codebase with further assumptions while trying to maintain structural integrity of what it previously built.

🎯

Logical, Not Syntactic

The code compiles. Types check. Basic tests pass. The failure is logical — reasonable choices about business rules, error handling, and domain constraints that are silently wrong.

Implementation Mechanism

The Hierarchical Instruction Model

CLAUDE.md files at every layer of the codebase. Context narrows as you descend. The agent reads breadcrumbs top-down — no assumptions required.

Principle 1

Context Narrowing

Root says "Protocol-based design." Domain says "services depend on Protocol interfaces." Services says "inject via constructor, declare as Protocol types." Each level adds specificity.

Principle 2

Instructions Are Authoritative

If the instruction file and existing code disagree, the instruction file wins. The agent flags the inconsistency — it doesn't inherit the inconsistency.

Principle 3

Living Documents

Instruction files evolve with the code. Version-controlled, git-blamed, reviewed like code changes. The institutional memory of architectural decisions.

The Theory

The Constraint Architecture

Constraints eliminate interpretation. Instructions leave room for it. In AI-Native development, constraints are the primary mechanism for controlling agent behavior.

Constraint Inheritance — click to expand

Lower levels tighten but never loosen higher-level constraints.

The llm.txt Specification

Unlike CLAUDE.md (how to work), llm.txt defines what tools and patterns are available. Structured for machine consumption — options, decision frameworks, and explicit constraints per domain.

Property

Structured, Not Narrative

Clear headings, explicit enumerations, decision trees with concrete criteria. Designed for agents, not prose readers.

Property

Options, Not Opinions

Presents what is available and when to use each option. The agent selects based on constraints, not training bias.

Property

Domain-Scoped

Each domain has its own llm.txt. Payments says nothing about auth. Auth says nothing about payments. No cross-contamination.

Quality Mechanism

The Test-Driven Constraint Model

In AI-native development, tests are not verification — they are boundaries. A failing test is a hard constraint the agent cannot cross.

Core Rule

Tests Before Implementation

Tests written after implementation validate assumptions — circular reasoning. Tests written before constrain the implementation space. This is not optional.

Core Rule

Test Immutability

Tests from Steps 2-3 may NOT be modified in Step 4. If implementation fails a test: fix the implementation or formally revise the specification. Never silently change the test.

Core Rule

Constitution, Not Rubber Stamp

When test and implementation conflict, the test wins. Tests encode the specification. The implementation must satisfy it — not the other way around.

The Framework in One Sentence

Define constraints explicitly at every level, verify adherence automatically at every step, and treat the instruction hierarchy and test suite as the primary governance mechanisms — because the model's judgment is not a reliable substitute for structural enforcement.
